# 断电重启问题排查指南

## 问题描述

设备在断电一段时间后重新上电时，有概率无法正常启动。MCU似乎没有正常工作，ST-Link无法连接，提示"Run loop exited due to a locked up core"。

## 可能的原因分析

### 1. 电源稳定性问题
- **TPS62933DRLR 3.3V输出不稳定**：断电后重新上电时，电源可能需要更长时间稳定
- **电源纹波或瞬态响应**：重新上电时的电源质量可能影响MCU启动
- **电容放电时间**：断电时间不够长，电容未完全放电导致启动异常

### 2. 时钟系统问题
- **HSI时钟启动时间**：内部高速时钟在断电重启后可能需要更长稳定时间
- **PLL锁定失败**：170MHz的高频PLL在某些条件下可能锁定失败
- **时钟切换时序**：从HSI切换到PLL的时序可能存在问题

### 3. 复位和启动序列问题
- **复位信号质量**：外部复位信号可能存在毛刺或时序问题
- **启动代码执行**：Flash中的启动代码可能因为某种原因无法正确执行
- **栈指针初始化**：栈指针可能指向无效内存区域

### 4. 外部设备干扰
- **I2C总线状态**：外部I2C设备可能处于异常状态，影响MCU启动
- **GPIO状态冲突**：某些GPIO引脚可能被外部电路拉到异常状态

## 改进措施

### 1. 增强的系统配置
已对系统配置进行以下改进：

```rust
// 更保守的时钟配置，增加稳定性注释
config.rcc.pll = Some(Pll {
    source: PllSource::HSI,
    prediv: PllPreDiv::DIV4,
    mul: PllMul::MUL85,
    divp: None,
    divq: None,
    divr: Some(PllRDiv::DIV2), // 170MHz
});
```

### 2. 启动延时和稳定性检查
- **系统初始化延时**：在main函数开始时增加200ms延时
- **I2C初始化延时**：在I2C初始化前增加50ms延时
- **硬件组件初始化延时**：在各硬件组件初始化间增加适当延时

### 3. 重试机制
- **SW2303初始化重试**：最多重试3次，每次间隔100-200ms
- **I2C通信超时设置**：设置1秒超时，提高可靠性

### 4. 看门狗保护机制
添加了独立看门狗（IWDG）来提高系统可靠性：

- **早期初始化**：在硬件初始化开始时就启用看门狗
- **5秒超时**：如果系统卡死超过5秒，自动复位
- **定期喂狗**：在主循环和关键初始化步骤中定期喂狗
- **启动保护**：防止启动过程中的死锁导致系统无响应

### 5. 诊断工具
创建了多个诊断工具：

```bash
# 基本功能诊断
make diagnostic
# 看门狗功能测试
make watchdog-test
```

## 排查步骤

### 步骤1：运行诊断工具
首先运行诊断工具确认MCU基本功能：

```bash
make diagnostic
```

如果诊断工具能正常运行，说明MCU硬件正常，问题在于复杂的初始化序列。

### 步骤2：检查电源质量
1. 用示波器检查3.3V电源的启动波形
2. 检查电源纹波是否在规格范围内
3. 测量电源上升时间和稳定时间

### 步骤3：检查复位信号
1. 检查NRST引脚的复位信号质量
2. 确认复位信号的时序符合STM32规格
3. 检查是否有外部复位电路干扰

### 步骤4：逐步启用功能
1. 先运行最简单的程序（如诊断工具）
2. 逐步添加I2C、显示、传感器等功能
3. 确定哪个组件导致启动失败

### 步骤5：检查外部设备
1. 断开所有外部I2C设备，只保留MCU
2. 逐个连接外部设备，确定问题设备
3. 检查I2C上拉电阻和信号质量

## 硬件建议

### 1. 电源改进
- 在3.3V输出端增加更大容量的电容（如100μF）
- 添加电源监控电路，确保电源稳定后再释放复位
- 考虑使用电源监控芯片（如TPS3813）

### 2. 复位电路改进
- 添加外部复位监控电路
- 增加复位延时电路，确保电源稳定后再释放复位
- 添加复位按钮用于手动复位

### 3. 时钟电路改进
- 考虑使用外部晶振代替内部HSI
- 添加时钟监控和故障切换机制

## 软件建议

### 1. 看门狗
考虑添加独立看门狗（IWDG）：
- 在系统启动后启用看门狗
- 在主循环中定期喂狗
- 看门狗超时时自动复位系统

### 2. 错误处理
- 添加更完善的错误处理和恢复机制
- 记录启动失败的原因到Flash
- 实现安全模式启动

### 3. 启动检查
- 添加启动自检程序
- 检查关键硬件组件状态
- 实现渐进式启动策略

## 测试建议

### 1. 重复测试
- 进行大量的断电重启测试
- 记录失败率和失败模式
- 测试不同的断电时间（1秒、10秒、1分钟等）

### 2. 环境测试
- 在不同温度下测试
- 测试不同的电源电压（如3.0V、3.3V、3.6V）
- 测试电源噪声对启动的影响

### 3. 长期稳定性测试
- 24小时连续运行测试
- 定期断电重启测试
- 监控系统运行状态

## 应急措施

如果问题持续存在，可以考虑以下应急措施：

1. **降低系统频率**：将PLL频率从170MHz降低到更保守的值
2. **简化启动序列**：移除非关键的初始化步骤
3. **增加启动延时**：大幅增加各阶段的延时时间
4. **使用外部看门狗**：添加外部硬件看门狗电路
