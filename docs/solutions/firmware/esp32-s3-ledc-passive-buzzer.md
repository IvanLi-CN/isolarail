## Context

The board uses an ESP32-S3-family MCU and routes the passive buzzer network
`BUZZER` to `GPIO7`. The firmware also drives the fan control network on
`GPIO1` with LEDC PWM. Buzzer tones are short event sounds and alarms, so the
firmware must keep CPU time available for runtime sampling, I²C, fan control,
USB JSONL, and network tasks.

For a passive buzzer, the MCU must generate the audio-frequency waveform. A
stable hardware PWM source is therefore the preferred driver.

## Symptoms

- GPIO half-period busy-waiting steals CPU time and can break unrelated runtime
  scheduling.
- Async GPIO toggling at every half-period avoids a tight busy loop, but still
  wakes the scheduler at audio frequency and produces jitter-prone tones.
- RMT can output finite pulse sequences, but tone playback needs sequence
  chunking or reloads, which adds interrupt and buffer work for a periodic
  waveform.
- Treating the fan as the exclusive owner of the whole LEDC peripheral hides
  remaining timer/channel capacity and pushes the buzzer toward weaker drivers.

## Root Cause

The buzzer needs a stable square wave; generating that wave in software
competes with the rest of the firmware. The first implementation constraint was
software ownership, not hardware capacity: the fan task took ownership of the
whole `LEDC` peripheral even though it only needed one timer and one channel.

ESP32-S3 LEDC supports PWM output through timer and channel configuration. On
ESP32-S3 all LEDC timers/channels are low-speed mode, so PWM setting changes
must be explicitly applied by software, but the waveform itself is generated by
hardware after configuration.

## Resolution

- Initialize the LEDC peripheral once during startup.
- Reserve distinct low-speed resources:
  - fan: `Timer0` + `Channel0` + `GPIO1`;
  - buzzer: `Timer1` + `Channel1` + `GPIO7`.
- Bind the buzzer channel to its timer once, with push-pull output and initial
  `0%` duty.
- For each note, update the buzzer timer frequency, then set channel duty to
  `50%` for the note duration.
- For rests, alarm gaps, and playback completion, set duty back to `0%` so the
  output remains silent and low.
- Use async timers only for note duration and gaps; never use per-half-period
  delays.

The firmware computes the LEDC timer divider for each note using the same model
as the HAL driver:

```text
divider = (source_clock_hz << 8) / (frequency_hz * 2^duty_bits)
```

For the current tone table, APB clock with 10-bit duty resolution keeps the
audio frequencies in range. If the tone table later adds much lower or higher
frequencies, re-check the divider range before accepting the new sounds.

## Guardrails / Reuse Notes

- Use LEDC/PWM for passive buzzer tones on ESP32-S3 unless LEDC resources are
  genuinely unavailable.
- Use RMT for protocol-like or non-periodic pulse trains, not as the default
  passive buzzer tone generator.
- Do not generate audio-frequency square waves with CPU busy-waiting or
  per-half-period async sleeps.
- Do not attach another variable-frequency output to the buzzer timer; changing
  `Timer1` frequency affects every channel bound to that timer.
- Keep fan and buzzer timer/channel assignments documented and fixed unless a
  resource map change is made deliberately.
- If an active buzzer replaces the passive buzzer, simple GPIO on/off may be
  sufficient because the sounder contains its own oscillator.
- For ESP32-S3 low-speed LEDC, remember to apply timer parameter updates after
  changing frequency-related registers.
